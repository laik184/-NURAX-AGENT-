English Specification (Main Document)
1. Project Context
NURA X is a Replit-clone AI web IDE. The frontend is a polished React + Vite SPA in client/ that mimics a full IDE (workspace, chat, file tree, terminal, preview, publishing). The backend in server/ already contains a sophisticated multi-agent system with intent detection, planning, code generation, validation, recovery, and memory phases — but only POST /api/agents/run is wired up. Most of the IDE-style endpoints the frontend talks to do not exist yet.

Your job is NOT to redesign anything. Your job is to:

Build the missing backend endpoints the frontend already calls.
Build a single Orchestration Controller that ties the existing agents together into a usable pipeline triggered from the UI.
Make the live streams (SSE + WebSocket) work end-to-end.
Persist project/session data in Replit Postgres.

4.2 The Orchestration Controller — server/orchestration/controller.ts
This is the brain. One class, clear responsibilities:

class OrchestrationController {
  // Entry: called by POST /api/run with { projectId, goal, mode }
  async runGoal(input: RunInput): Promise<RunHandle>
  // Streams progress to all subscribers (WS rooms keyed by runId)
  private emit(runId, event: AgentEvent)
  // Hybrid pipeline:
  //   1. safety-gate (fixed)
  //   2. routing (intent-detector → domain-mapper → agent-selector → confidence-scorer)
  //      → if confidence < threshold, ask LLM "decision-engine" to override
  //   3. planning (structure-planner)
  //   4. generation (prompt-builder → code-writer → output-validator)
  //      → write CodeFile[] into .sandbox/<projectId>/ via filesystem.service
  //      → enqueue diffs into diff_queue table
  //   5. execution (optional: shell.exec inside sandbox dir)
  //   6. feedback + memory (persist agent_run + agent_events)
  // Each step emits AgentEvent → broadcast to WS subscribers + persist
}

Key rules:

All agent invocations go through this controller — never call agents from routes directly.
Every state change becomes an AgentEvent row + WS broadcast.
Routing-agent confidence < 0.6 → call decision-engine to pick alternate strategy.
Failures → call recovery agent before giving up.

11. Final Note to the Implementing Agent
The frontend is a demo shell that already speaks a clear protocol. Your job is to make the backend honor that protocol while leveraging the powerful agent system that already exists. The existing pipeline orchestrator at server/agents/core/pipeline/orchestrator.ts is your starting reference — don't rewrite it, wrap and call it from the new OrchestrationController. Treat existing agents as black-box capabilities and stitch them together via the event bus.

When in doubt: ship the smallest thing that turns a frontend interaction into a streamed backend response, then layer.
