1 · Scope & Method
A deep scan of client/src/ (~80 components, hooks, services, contexts) was matched against server/ (routes, controllers, streams, orchestration). For every UI interaction we extracted: location → trigger → function → method/URL → payload → expected response. Each call was then classified against the running server.

8 · Remaining / Deferred
These endpoints respond successfully but with intentional safe stubs because the underlying capability isn't implemented in this repo yet:

Endpoint	Stub behavior	Why
POST /api/files/upload	{ ok: true, data: { uploaded: 0 } }	Multipart parser (e.g. multer) not installed. Plug in when needed.
POST /api/git/{init,commit}	Always success with stub SHA	The Replit sandbox doesn't expose git binary access here.
POST /api/file/undo	{ undone: false }	A per-file undo stack isn't tracked yet (only diff-queue history is).
POST /api/file/conflict-check, /api/fs/conflict-check, /api/fs/conflict-details	{ conflict: false }	Server-side conflict detection requires a content-hash store; UI gracefully proceeds when no conflict.
POST /api/solopilot/prepareConflict	{ conflict: false, blocks: [] }	Same reason.
All of these can be promoted to real implementations without touching the UI by swapping the handler bodies in server/routes/compat.routes.ts.

9 · Result
Every UI button, form, hook, service call, WebSocket and EventSource in client/src/ now has a backing endpoint that returns the normalized { ok, data?, error? } envelope. Logic flows through the OrchestrationController → executePipeline → agents pipeline. Sandbox enforcement is mandatory on every file route. The frontend was not modified.
